# 报告模板：code-review-multi

输出 Markdown 文件严格套用以下结构（多仓库循环重复仓库区块，用 `---` 隔断）：

```markdown
# 📦 仓库名称：[REPOSITORY 名]

**分支对比**：`[源分支]` -> `[当前分支]`

## 📝 变更概述

[简明扼要总结该仓库的主要代码变动点]

## 🚨 深度审查意见

*(三大类别必须全部输出。若某类无发现，写：`本次变更未发现此类问题。`，不得省略类别标题。)*

### 🚫 严重问题 (Critical)

**[序号]、**

* **问题描述**：[精准描述缺陷原因]
* **潜在影响**：[描述后果]
* **位置**：第 L[起始行号] - L[结束行号] 行
* **当前分支和原分支改动对比**：

  ```[language]
  // 原分支代码
  [完整提取原分支代码块]

  // 当前分支改动代码
  [完整提取当前分支改动的代码]
  ```

* **修复对比**：

  ```[language]
  // ❌ 原始代码
  [完整提取包含问题的原始代码块]

  // ✅ 修复代码
  [提供符合 Clean Code、高性能的重构代码]
  ```

* **修复重点**：[简述修复后的核心优势]

### ⚠️ 改进建议 (Standard)

**[序号]、**

* **建议点**：[描述建议]
* **理由**：[说明为什么这样改更好]
* **位置**：第 L[起始行号] - L[结束行号] 行
* **当前分支和原分支改动对比**：

  ```[language]
  // 原分支代码
  [完整提取原分支代码块]

  // 当前分支改动代码
  [完整提取当前分支改动的代码]
  ```

* **修复对比**：

  ```[language]
  // ❌ 原始代码
  [完整提取包含问题的原始代码块]

  // ✅ 修复代码
  [提供符合 Clean Code、高性能的重构代码]
  ```

* **修复重点**：[简述修复后的核心优势]

### 💡 优雅重构 (Refactoring)

**[序号]、**

* **位置**：第 L[起始行号] - L[结束行号] 行
* **当前分支和原分支改动对比**：

  ```[language]
  // 原分支代码
  [完整提取原分支代码块]

  // 当前分支改动代码
  [完整提取当前分支改动的代码]
  ```

* **修复对比**：

  ```[language]
  // ❌ 原始代码
  [完整提取包含问题的原始代码块]

  // ✅ 修复代码
  [提供符合 Clean Code 原则的重构代码]
  ```

* **修复重点**：[简述修复后的核心优势]

## 🏁 总结评述

* **整体评价**：[公式：10 - (严重×3) - (改进×0.5) - (重构×0.2)，最低 1 分，保留一位小数]
* **核心风险**：[一句话总结最需要关注的改动点]

---

*(多仓库时循环重复上述区块)*

# 📊 全局执行汇总

*(置于文档最末尾)*

* **扫描总数**：[TOTAL_SCANNED_REPOS]
* **审查总数**：[实际产生 Diff 报告的仓库数]
* **跳过总数**：[未生成报告的仓库数]
* **跳过详情**：
  * [仓库名]：[SKIP 日志中括号内的具体原因]
```
