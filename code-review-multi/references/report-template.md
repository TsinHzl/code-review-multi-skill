# 报告模板：code-review-multi（梯度化版本）

## 仓库区块骨架（每个有效仓库循环重复，用 `---` 隔断）

```markdown
# 📦 仓库名称：[REPOSITORY 名]

**分支对比**：`[源分支]` -> `[当前分支]`

## 📊 变更统计

| 指标 | 数值 |
|------|------|
| 涉及文件数 | [N] 个（已由 env 脚本剔除自动生成文件） |
| 新增行数 | +[N] |
| 删除行数 | -[N] |
| 净变更行数 | [±N] |
| 涉及提交数 | [N] 个 |

**主要变更文件：**

| 文件路径 | 新增 | 删除 | 变更类型 |
|---------|------|------|----------|
| `[路径]` | +[N] | -[N] | [新增/修改/删除/重命名] |

**一句话摘要：** [本次改动 N 个文件，主要涉及 X 的 Y]

## 📝 变更概述

[简明扼要总结该仓库主要代码变动点]

## 🚨 深度审查意见

[按 Critical → Standard → Refactoring 顺序输出。**任一类零问题则整段省略**（含标题与计数），不写"未发现"。]

## 🏁 总结评述

* **整体评价**：[公式：10 − 严重×3 − 改进×0.5 − 重构×0.2，最低 1.0 分，保留一位小数。格式：X.X 分（严重×N，改进×N，重构×N）]
* **核心风险**：[一句话]

---
```

## 三类问题字段梯度（严格遵守）

### 🚫 严重问题 (Critical) — 完整 5 字段（保留双代码块对比）

````markdown
### 🚫 严重问题 (Critical) - 共 [N] 个

**[序号]、**

* **问题描述**：[精准描述缺陷原因]
* **潜在影响**：[后果]
* **位置**：第 L[起] - L[止] 行
* **修复对比**：

  ```[language]
  // ❌ 原始代码
  [完整提取包含问题的原始代码块]

  // ✅ 修复代码
  [Clean Code 重构代码]
  ```

* **修复重点**：[一句话核心优势]
````

### ⚠️ 改进建议 (Standard) — 精简 3 字段（保留 ❌/✅ 双代码块）

````markdown
### ⚠️ 改进建议 (Standard) - 共 [N] 个

**[序号]、** [一句话合并建议点与理由]

* **位置**：第 L[起] - L[止] 行
* **修复对比**：

  ```[language]
  // ❌ 原始代码
  [完整提取包含问题的原始代码块]

  // ✅ 修复代码
  [Clean Code 重构代码]
  ```
````

### 💡 优雅重构 (Refactoring) — 最简 2 字段（行内位置 + ❌/✅ 双代码块）

````markdown
### 💡 优雅重构 (Refactoring) - 共 [N] 个

**[序号]、** L[起]-L[止] — [一句话建议]

  ```[language]
  // ❌ 原始代码
  [完整提取包含问题的原始代码块]

  // ✅ 修复代码
  [Clean Code 重构代码]
  ```
````

```markdown
| 累计净变更行数 | [±N] |

* [仓库名]：[SKIP 日志中括号内的具体原因]
```
